# 武裝覺醒系統 — 裝備頁整合設計

**Date:** 2026-04-26
**Status:** Approved for implementation planning
**Scope:** 在裝備詳情頁 (`/items/[id]`) 嵌入「覺醒升階」區塊，玩家不需離開頁面就能看到該裝備可走的覺醒路徑、成本、機率、累積花費。

---

## 1. Overview

`strong_formula` 表是「裝備覺醒系統」的設定資料：每筆 row 描述某類裝備（依**世代 + 部位**分類）在某階段（`+1` ~ `+20`）需要的金錢、覺醒符、成功率，以及該階提供的屬性加值。

本設計把這份資料整合進**裝備詳情頁底部**，作為一個獨立區塊，沿用 `monsters/[id]` 掉落區塊的視覺節奏。**不**做獨立瀏覽頁（純配方表瀏覽價值低）；**不**做互動計算器（資料夠靜態，純表格 + 累積欄就足夠）。

## 2. 資料對應規則

### 2.1 世代代號（依 `items.level` 分區）

| `items.level` | 世代 prefix | 主要覺醒符 |
|---|---|---|
| 1 ~ 39 | `20` | 一星 |
| 40 ~ 59 | `40` | 二星 |
| 60 ~ 79 | `60` | 三星 |
| 80 ~ 99 | `80` | 四星 |
| 100 ~ 119 | `100` | 五星 |
| 120 ~ 139 | `120` | 六星 |
| 140 ~ 159 | `140` | 七星 |
| 160 ~ 179 | `160` | 八星 |
| 180 ~ 199 | `180` | 九星 |
| 200+ | `200` | 十星 |

> 第一段是 1~39（不是 1~19），與一星覺醒符的 `summary` 一致。

### 2.2 部位代號（依 `items.type`）

直接相等對應，無需轉換：

| `items.type` | prefix 部位 |
|---|---|
| 鞋 | 鞋 |
| 衣 | 衣 |
| 甲 | 甲 |
| 盾 | 盾 |
| 帽 | 帽 |
| 手套 / 手甲 | （未進入覺醒系統，暫不顯示） |
| 座騎 | 座騎 |
| 中飾 / 左飾 / 右飾 / 背飾 | 中飾 / 左飾 / 右飾 / 背飾 |
| 單手武器（劍/刀/匕首/扇/拂塵/拳刃/雙劍/暗器/棍） | 單手武器 |
| 雙手武器（雙手刀…） | 雙手武器 |
| 法杖 | 法術武器 |

實際 `items.type` 比 prefix 部位多很多分類（如「劍」「拳刃」），所以需要一張靜態 `itemType → slotPrefix` 對照表（見 §6.1）。**找不到對應就不顯示區塊**（不報錯）。

### 2.3 完整的 prefix 集合

114 組合法 prefix（見 `scratch-strong-formula.js` 探索結果），由 §2.1 × §2.2 笛卡兒積生成，扣除實際資料表沒出現的（如 `20雙手武器`、`200中飾`）。實作時直接 `WHERE name = '{prefix}+N'` 查詢，找不到視為不適用。

### 2.4 過濾規則（資料載入時）

```sql
SELECT *
FROM strong_formula
WHERE name LIKE '{prefix}+%'
  AND name NOT LIKE '%"%'   -- 排除跨階煉化 row（如 20鞋20"8）
ORDER BY CAST(SUBSTR(name, LENGTH('{prefix}') + 2) AS INTEGER), bonus_type
```

跨階煉化（含 `"`）是另一個系統（直接從 +N 跳到 +M），第一期不處理。

## 3. 屬性加值結構

### 3.1 單屬性 vs 多屬性

`strong_formula` 大多數 prefix **只有單一屬性**，例如：

- 鞋/甲/帽/盾/座騎/單手武器/雙手武器/各部位（除 200 系特例）→ 單屬性
- 一般慣例：防具系（鞋/甲/帽/盾/座騎/背飾）= DEF；衣/中飾/左飾/右飾 = MDEF；單手/雙手武器 = ATK；法術武器 = MATK

### 3.2 200 系特例（多屬性）

只有以下 3 個 prefix 提供 5 種屬性可選：

- `200鞋` → DEF / ATK / MATK / DODGE / HP
- `200甲` → DEF / ATK / MATK / DODGE / HP
- `200帽` → DEF / ATK / MATK / DODGE / HP

UI 對這 3 個 prefix 展開成 5 欄並列；其餘單欄。

### 3.3 已知資料 quirk

部分 prefix（`100右飾` 全 20 階、若干 `商城xx`）在同 `(name, bonus_type)` 有 **2 條 row**，分為兩種狀況：

1. **真複製**（例：`100右飾+11` 兩條 MDEF row 完全一樣）
2. **雙路徑**（例：`100右飾+1` — 五星覺醒符 +1 vs 九星覺醒符 +2，金錢/材料/加值都不同，是「便宜舊路徑」對「貴新路徑」的兩個遊戲選項）

實作策略：**同 stage 同 `bonus_type` 取 `material_core_id` 最大的那 row**（取新階梯路徑、加值較高），其餘丟棄。`100右飾+1` → 取 `26967` (九星，bonus_value=2, money=60000)。商城 prefix 因不會被一般 `(level, type)` 對應規則查到，自然不渲染。

## 4. UI 設計

### 4.1 位置與結構

裝備詳情頁底部新增 `<section>`「覺醒升階」，置於現有屬性表 / 來源資訊之後。沿用既有的 shadcn `Card` + `Table` primitives。

```
┌─────────────────────────────────────────────────┐
│ 覺醒升階                                          │
│                                                  │
│ ［說明文字：適用範圍 + 材料 + 機制要點］           │
│                                                  │
│ ┌────────────────────────────────────────────┐  │
│ │ 階段 │ 加值 │ 此階金錢 │ 此階符 │ 成功率 │  │
│ │      │      │         │       │       │  │
│ │      │      │         │ 累積金錢│累積符│  │
│ ├──────┼──────┼─────────┼───────┼───────┤  │
│ │ +1   │ DEF+2│ 100,000 │十星×1 │100%   │  │
│ │ ...  │      │         │       │       │  │
│ │ +18  │ DEF+36│9M      │超越×1 │10%   │  │
│ │ +20  │ DEF+40│12M     │超越×1 │3%    │  │
│ └────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 4.2 表格欄位

| 欄 | 內容 | 備註 |
|---|---|---|
| 階段 | `+1` ~ `+20` | |
| 屬性加值 | 單欄（一般）或 5 欄並列（200 鞋/甲/帽） | 從 `bonus_type` 翻譯成中文 + `bonus_value` |
| 此階金錢 | `money` | `Intl.NumberFormat` 格式化 |
| 此階符 | `material_core_id → 名稱` × `material_core_amount` | 對應到 items 表查名稱 |
| 此階成功率 | `success_prob / 10000` % | 以 1 位小數呈現 |
| 累積金錢（最佳） | `Σ money_i` | 假設一發過 |
| 累積金錢（期望） | `Σ (money_i / p_i)` | 假設失敗不掉階 |
| 累積符 | 各覺醒符的最佳累計次數 | 例：`十星×10, 突破×5, 超越×3` |

> 「累積金錢（最佳）」與「累積金錢（期望）」可在同一儲存格用兩行呈現（最佳 / 期望），減少欄寬負擔。

### 4.3 Milestone 視覺強調

對 `+10`、`+15`、`+18`、`+20` 四個常見停損點加底色或粗框（class: `bg-muted/50`），讓玩家一眼看到關鍵點。理由：
- `+10`：基礎覺醒終點，符種類即將切換為「突破」
- `+15`：突破階段終點，進入「超越」
- `+18`：玩家常見實戰停損（CP 值高點）
- `+20`：理論上限

### 4.4 RWD 策略

- **Desktop (≥768px)**：完整表格
- **Mobile**：表格水平捲動（沿用 `monsters/[id]` 掉落表的 `overflow-x-auto` 模式），不改 layout 為 stacked card，避免 20 階變成 20 張卡的長頁

### 4.5 Footer 說明區塊

表格下方加一個 collapsible section（用 base-ui `Collapsible`，預設展開）：

> **覺醒機制速查**
> - **失敗保護**：`+1`~`+5` 不毀裝、`+11`~`+17` 不掉階。
> - **掉階規則**：`+17` 衝 `+18` 失敗會降回 `+16`；`+18`~`+20` 失敗依序掉階。
> - **退階防護令**（商城）：`+18` 需 140 張、`+19` 需 170 張、`+20` 需 200 張可保不掉階。**本表「累積金錢（期望）」未計入此成本**。
> - **幸運值**：突破失敗會累積幸運值，提升下次突破機率，成功後重置。
> - **覺醒符煉化**：七星以上可由 3 張次階符煉化取得；超越覺醒符可由 3 張突破覺醒符煉化取得。

## 5. 累積期望成本公式

### 5.1 公式

對階段 `n`，期望嘗試次數 = `1 / p_n`；該階段期望金錢消耗 = `money_n / p_n`。

從 `+0` 累積到 `+N` 的期望總金錢：

```
E[cost(N)] = Σ_{i=1..N} (money_i / p_i)
```

**前提假設**：失敗不掉階（消耗符 + 錢，裝備停在當前階）。

### 5.2 已知偏差

`+18`~`+20` 的失敗會掉階，實際期望成本比公式高（每次失敗會浪費上一階的成功成本）。本期**不**精算此案；UI 在 footer 註明「實際更高，可用退階防護令避免」。第二期可加上「含退階防護令成本」選項。

### 5.3 累積符數量（最佳）

每階消耗 `material_core_amount` 張對應符。最佳累積就是 `Σ amount_i` 分組到 `material_core_id` 名稱。

```
+1~+10: 十星 ×10
+11~+15: 突破 ×5
+16~+20: 超越 ×5
```

（200 系；其他世代換對應星符）

## 6. 實作骨架

### 6.1 資料層 — `src/lib/queries/awakening.ts`

```typescript
// 世代代號 mapping
export function levelToGenPrefix(level: number): string | null {
  if (level >= 200) return '200';
  if (level >= 180) return '180';
  if (level >= 160) return '160';
  if (level >= 140) return '140';
  if (level >= 120) return '120';
  if (level >= 100) return '100';
  if (level >= 80) return '80';
  if (level >= 60) return '60';
  if (level >= 40) return '40';
  if (level >= 1) return '20';  // 1~39
  return null;
}

// type → slot prefix（手寫 mapping）
export function itemTypeToSlotPrefix(type: string): string | null {
  // 防具
  if (type === '鞋' || type === '衣' || type === '甲'
      || type === '盾' || type === '帽' || type === '座騎') return type;
  if (['中飾', '左飾', '右飾', '背飾'].includes(type)) return type;
  // 武器歸類（單手武器 prefix 涵蓋大多數武器，依實際資料表只有 單手武器 / 雙手武器 / 法術武器 三類）
  const oneHand = ['劍', '刀', '匕首', '扇', '拂塵', '拳刃', '雙劍', '暗器', '棍'];
  if (oneHand.includes(type)) return '單手武器';
  if (type === '雙手刀') return '雙手武器';
  if (type === '法杖') return '法術武器';
  // 手套 / 手甲：覺醒符 summary 把它們歸類成獨立部位但 strong_formula 沒有對應 prefix，視為不適用
  return null;
}

export function getAwakeningPath(item: Item): AwakeningStage[] | null { ... }
```

### 6.2 UI 層 — `src/components/items/awakening-section.tsx`

Server component，接 `Item`，內部呼叫 `getAwakeningPath`，找不到回傳 `null`（不渲染）。

### 6.3 嵌入點 — `src/app/items/[id]/page.tsx`

在現有版面最下方加 `<AwakeningSection item={item} />`。

## 7. Out of Scope（第二期保留）

1. **商城限定 prefix**（`商城40帽`、`120商城座騎` 等）：formula.level 標記怪異、樣本量小，需要逐一驗證對應規則
2. **舊式合成**（綿羊/青蛇/獵人/土撥鼠/蜘蛛/蝙蝠強化裝備）：是另一個系統（覺醒符前的 真元 系統），第一期不混進來
3. **跨階煉化** row（含 `"` 的）：是「+N 直接跳 +M」的另一機制，第一期不處理
4. **退階防護令** 成本納入累積計算
5. **幸運值** 對成功率的修正（資料表沒這資訊，純機制）
6. **互動式計算器**（選目標階 + 屬性 → 算累積）
7. **`1400商城座騎` typo 5 筆**：跟 §1 的商城限定一起處理

## 8. 邊界與風險

| 風險 | 處理 |
|---|---|
| 玩家裝備找不到對應 prefix（如 手套、未知 type） | 不渲染區塊，無錯誤 |
| 資料表特定 prefix 不全（如 `20雙手武器` 不存在） | 找不到視為不適用，不報錯 |
| `100右飾+1` 重複 row | `(name, bonus_type)` dedupe |
| 商城裝備混進 `200` 系列邏輯（誤判） | 第一期不偵測商城屬性，按通用規則處理；若實際資料導致誤顯示再加白名單 |
| 200 鞋/甲/帽 5 屬性表格在 mobile 太寬 | `overflow-x-auto`，跟其他寬表一致 |

---

## 9. 驗證準則

1. **隨機抽樣 5 件不同 level 的裝備**（如 level=20 的鞋、level=120 的衣、level=200 的甲），手動對照覺醒表內容跟資料庫吻合
2. **不適用裝備**（如 `究極黑布鞋` level=7 屬於 `20鞋` prefix，應正常顯示；藥品/寶箱類則不渲染區塊）
3. **累積金錢公式**：用 `200鞋` 跑一遍，最佳累積應為 `103,900,000`；期望累積（失敗不掉階假設下）約為 `1,174,562,698`（+20 那階單獨期望 ≈ 4 億，主導總和）
4. **Milestone 視覺**：+10/+15/+18/+20 列要肉眼可辨
5. **RWD**：Mobile (375px) 表格能水平捲動、不破版
